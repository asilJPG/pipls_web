## 2. Структура базы данных (Supabase)

### Таблица `bot_actions`
Хранит действия, логируемые ботом. Основной тип действий для финансовой аналитики — `action_type = 'cash'` (кассовые отчеты).
* **Поля**:
  - `id`: уникальный идентификатор записи (ID отчета).
  - `created_at`: дата создания записи (дата смены, например, `2026-05-12T12:00:00+05:00`).
  - `action_type`: тип действия (`cash` для кассовых отчетов).
  - `document_number`: номер документа из iiko.
  - `details`: JSONB-поле с деталями отчета.
* **Структура `details`**:
  ```json
  {
    "selected_date": "YYYY-MM-DD",
    "iiko_cash": 15000000.0,
    "reported_cash": 15000000.0,
    "difference": 0.0,
    "employee_wages": [
      {
        "name": "Бойматов Шохрух",
        "wage": 300000.0,
        "employeeId": "uuid-сотрудника"
      }
    ]
  }
  ```

### Таблица `cash_reports`
Хранит агрегированные отчеты по кассам.
* **Поля**:
  - `id`: идентификатор отчета.
  - `created_at`: дата/время создания.
  - `reported_cash`: сданные кассиром наличные.
  - `iiko_cash`: сумма наличных по данным iiko.
  - `difference`: расхождение (`reported_cash - iiko_cash`).

### Таблица `bot_users`
Хранит информацию о пользователях/сотрудниках и их правах доступа.
* **Добавленные поля (Оптимизация трекинга активности)**:
  - `last_login_at`: Timestamp последнего успешного входа пользователя в систему.
  - `last_login_method`: Способ последнего входа (например, `password` или `passkey`).

## 3. Реализованный функционал

### 💰 Аналитика Сейфа (Вкладка «Сейф»)
* **Расположение**: Компонент `LocmacoApp.jsx`
* **Отображаемые метрики**:
  1. **Остаток в сейфе (Итоговый баланс)** [Зеленый цвет]: Общая сумма всех сданных кассирами наличных за все время минус все административные расходы из сейфа.
  2. **Сдано наличными за период** [Синий цвет]: Сумма наличных, сданных за выбранный интервал дат.
  3. **Расходы из сейфа за период** [Красный цвет]: Административные и хозяйственные списания из сейфа за выбранный интервал дат.

### 👥 Аналитика зарплат сотрудников (Вкладка «Заработная плата сотрудников»)
* **Эндпоинт**: `/api/iiko/analytics/wages?from=YYYY-MM-DD&to=YYYY-MM-DD`
* **Функционал**:
  - Расчет общей суммы выплаченных зарплат и среднего размера выплат за смену.
  - Таблица ежедневных отчетов с суммами и числом сотрудников.
  - Детализация по каждому сотруднику и его зарплате за выбранный день.
  - **Индикатор последних данных**: Отображает плашку с датой последнего внесенного отчета о зарплатах (например, `📅 Последние данные внесены за: 2026-06-10`) и выводит подсказку, если за выбранный период данных еще нет.

### 📝 Ввод зарплат при закрытии смены (Форма кассира)
* **Эндпоинт активных сотрудников**: `/api/iiko/employees/active` — собирает явки сотрудников из iiko Server и сопоставляет их с тарифными ставками из iikoWeb.
* **Форма**: При создании кассового отчета кассир видит список сотрудников на смене и может отредактировать их начисленные за день зарплаты. Данные сохраняются в массив `details.employee_wages`.

### 📂 Интерактивный отчет P&L и детализация расходов
* **Основной эндпоинт**: `/api/iiko/analytics/pl?from=YYYY-MM-DD&to=YYYY-MM-DD` (загружает агрегированные категории мгновенно).
* **Эндпоинт деталей**: `/api/iiko/analytics/pl/details?category=...&from=...&to=...` (загружается on-demand при клике).
* **Функционал**:
  - Анализ выручки, себестоимости продаж и операционных расходов за выбранный период на круговой диаграмме `PieChart`.
  - **Интерактивность**: Клики по секторам самой круговой диаграммы (SVG Pie Chart) отключены. Модальное окно детализации открывается **только** при нажатии на легенду/строки списка категорий. Все транзакции с одинаковым номером документа группируются в одну строку с общей суммой, а в «Доп. информации» выводятся первые 3 позиции. Для категории «Зарплата» все автоматические транзакции по сменам (без номера документа) полностью отфильтровываются, отображая только официальные документы выплат (например, `0173`).

### 🔒 Ограничение доступа к разделу «Сотрудники»
* **Ограничение**: Доступ к вкладке «Сотрудники» во фронтенде и её API-эндпоинтам (`/api/iiko/employees/*`) ограничен строго ролью `admin`.
* **Суть**: Пользователи с ролями `manager` и `director` больше не могут просматривать список сотрудников и их коды доступа (пароли) как на UI, так и напрямую через REST-запросы к бэкенду.

### 🔑 Двойное логирование входов (Аудит логов)
* **Принцип работы**: 
  - При входе по коду/паролю или биометрии Face ID/Touch ID система одновременно обновляет поля последнего входа в `bot_users` (для быстрого отображения в UI) и создает новую строку лога в `bot_actions` (с типами `LOGIN_PASSWORD` или `LOGIN_PASSKEY`), чтобы сохранялся полный аудит активности пользователей.

## 4. Импорт исторических данных по ЗП

Была произведена полная выгрузка и импорт данных за период **с 14 апреля по 10 июня 2026 года (всего 58 дней)**.

* **Источник**: Таблица Excel `/Users/asil/Downloads/yandex/The Lokmaco Fergana -ЗП таблица  (1).xlsx`
* **Процесс импорта**:
  1. Скрипт очистки `clear_all_wages.mjs` обнулил все старые данные в базе.
  2. Скрипт `parse_all_v3.py` разобрал листы Excel («Апрель», «Май», «Июнь») с учетом разбивки сотрудников по категориям (Бар, Кухня, Мойка и т.д.) и отсечением строк «Итого».
  3. Скрипт `import_spreadsheet_wages.mjs` записал корректные массивы зарплат во все 58 отчетов в базе данных.

### Карта канонических имен и исправление ошибок
Для предотвращения дублирования сотрудников из-за опечаток в Excel настроено автоматическое каноническое сопоставление имен к UUID:
* `Абдукодилов Ш` ➔ `Абдукодиров Ш`
* `Абдукодилов Шахзод` ➔ `Абдукодиров Шахзод`
* `Кидирбеав С` ➔ `Кидирбоев С`
* `Турдалиева Нилуфар` ➔ `Турдалиева Налуфар`
* `Бойматов Шихрух` ➔ `Бойматов Шохрух`
* `Азамхонов Аббосхон` / `Азамов/Аббосхон` ➔ `Аббосхон`

Карта соответствия имен и UUID сохранена в файле `name_to_id_map.json` в рабочей директории Gemini.
